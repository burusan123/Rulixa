# Rulixa Full Spec

## 0. 位置づけ

この文書は `Rulixa` の現時点の正本です。
Phase 1 の実装仕様や個別解析対象よりも上位にあり、製品の目的、価値、販売前提、アーキテクチャ上の不変条件を定義します。

## 1. 製品定義

`Rulixa` は、設計知・依存・制約・安全要件を継続生成し、AI に渡す入力を正規化し、PR レビューと監査に使える成果物へ変換するためのローカル基盤です。

生成対象は次です。

- `Contracts`
- `Index / Map`
- `Context Pack`
- `Security Gate` に基づく安全な実行制御

Context Pack は重要な一部ですが、製品全体の中心価値はそれだけではありません。
中心は、設計知を継続生成し、差分としてレビュー・監査できる状態を作ることです。

## 2. 主な価値

### 2.1 AI 入力の正規化

- 変更対象ごとに必要最小限の文脈を決定的に生成する
- 人や入口による文脈のぶれを減らす
- 暗黙仕様を成果物として固定する

### 2.2 PR レビュー短縮

- 変更に関係する契約と依存だけを差分で読める
- レビュー時に「何が前提で、何が影響範囲か」を短時間で把握できる
- AI レビューでも人間レビューでも同じ素材を使える

### 2.3 監査証跡

- いつ、どの入力から、どの成果物が生成されたかを追える
- secret 対策や危険操作制御を監査可能な形で残せる
- 設計変更の意図と差分を継続的に残せる

### 2.4 CI 継続生成

- リポジトリ変更に追従して成果物を再生成する
- ドキュメントと実装の乖離を継続的に観測する
- Pack だけでなく Contracts / Index も継続更新する

### 2.5 ドリフト検知

- 実装、設定、契約、成果物のズレを比較できる
- 生成結果の差分から設計ドリフトを検知する

## 4. Phase 1 の意味

Phase 1 は製品全体を定義するものではなく、最初の具体攻略対象です。
現在は `Windows` 上の `WPF + .NET 8` を選び、AI 入力最適化と最小 Context Pack 生成を先に実戦投入します。

これは製品の本質が WPF にあるからではなく、最短で検証できる具体対象だからです。

## 5. アーキテクチャ不変条件

- Core に価値を置く
- Front は薄く保つ
- Plugin は具体解析を担い、Core を汚さない
- 外側から内側へ依存する
- ローカルで完結する
- 同じ入力から同じ出力を返す

## 6. 販売前提

現時点の販売前提は次です。

- 年額課金
- `49,800円 / 席 / 年`
- 最低 3 席
- チーム利用前提
- PR レビュー、監査、CI 運用への組み込みを前提にする

個人買い切りは現時点の正本から外します。
販売資料、ライセンス実装、README はこの前提に揃えます。

## 7. セキュリティとライセンス

- secret 混入防止
- 許可外読み取りの抑止
- 危険操作の制御
- ライセンス状態の検証

ただし、製品価値の本体はライセンス保護ではなく、成果物生成と運用基盤にあります。

## 8. 成功条件

- AI が Pack だけで変更開始しやすくなる
- PR レビューで差分理解が速くなる
- 監査で設計知の根拠を辿れる
- CI で成果物を継続再生成できる
- ドリフトを比較可能な差分として見つけられる
