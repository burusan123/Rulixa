この1ファイルは、本プロジェクトの背景からゴール、アーキテクチャ、Phase計画、セキュリティ、販売・配布、運用までを **矛盾なく一気通貫で理解できる“正本”**としてまとめたものです。  
個別詳細は `docs/` 配下の各ドキュメントに分割して存在しますが、最終判断は本ファイルが優先されます（差分が出たら本ファイルに寄せて更新する）。

## 0. 目的（北極星の要約）

本プロジェクトの北極星は、**AI駆動開発の「トークン不足／暗黙仕様／巨大コードベース」を、誰の環境でも再現可能に“攻略可能な作業単位”へ変換する**ことです。

そのために、リポジトリから次を **決定的（同入力→同出力）** に生成し、AIに渡す“最小コンテキスト”を標準化します。

- **契約（Contracts）**: UI/API/データ/IaC/外部依存の前提・入出力・禁止事項を抽出し固定
- **索引（Index/Map）**: 依存関係・影響範囲・参照関係を機械的に可視化
- **Context Pack**: `entry + goal + budget` から最小ファイル束＋契約＋制約を組み立て
- **安全ガード（Security Gate）**: secret混入防止、読み取り範囲制限、危険操作の抑止

AIは「全文を読む」ではなく、**契約＋差分＋検証ログ**で局所問題を解く運用に寄せます。

（詳細は `docs/polaris.md`）

## 1. 背景（なぜ必要か）

### 1.1 AI駆動コーディングの限界

- コード量が大きいほど、LLMは仕様・制約・依存の全体像を保持できず、誤修正や横断依存による事故が増える
- ライブラリ/クラウド/運用の制約が複雑なほど、文章での注意喚起は破綻する
- “一発生成”は現実的ではなく、**テスト/コンパイラ/リンタ**を審判にした反復が必須

### 1.2 ブレイクスルーの方向性

RAG/ベクトル検索は有効だが必須ではなく、まずは次が本質的に効く。

- **境界設計**でAIが触る面積を小さくする
- 暗黙仕様を **契約（Contract）**に正規化する
- 制約を **実行可能なチェック（lint/型/テスト）**に落とす
- 失敗ログを **分類・圧縮**して反復修正を成立させる

## 2. 対象領域と優先順位（主戦場：GCP）

### 2.1 主戦場

- **GCPサービス／IaC／運用設定**
  - Cloud Run / PubSub / Workflows / IAM / Terraform / Cloud Build など

### 2.2 次点（多言語）

- **C# / TypeScript / Python / Rust**

### 2.3 入口（フロント）

- **VS Code（Cursor含む）**を主導線

## 3. OS方針（Phase 1〜）

### 3.1 Phase 1の基本方針

- **Windows: フル機能**
  - GCP/IaC + C#解析強め
- **macOS/Linux: ライト機能**
  - GCP/IaC中心 + **C#は軽量解析のみ**
  - 軽量解析の範囲: `using / 公開API / 簡易依存`
  - 型解決100%やビルド再現は保証しない（必要なら将来強化）

### 3.2 将来方針

- macOS/LinuxでのC#解析は、需要と効果に応じて段階的に深くする
- ただし「フォールバック可能（軽量で動く）」が不変条件

## 4. 製品形態（売れる構造）

### 4.1 “価値の中心”の置き方

販売と再配布抑止の観点から、価値はCore（バイナリ）に集約する。

- **Front（VS Code拡張）**: UI/UX、ワークフロー統合。薄く保つ
- **Core（ローカルバイナリ）**: 解析・生成・ルール検査・ライセンスチェック（有償価値の中心）
- **License API（GCP）**: ライセンス検証・端末管理のみ（コードや成果物は受け取らない）

### 4.2 再配布禁止の現実解

- 100%の技術的防止は不可能なので、**EULA（法務）＋ライセンス検証（技術）**で抑止・検知する

---

## 5. ライセンス仕様（個人買い切り：確定）

### 5.1 ルール

- **個人買い切り**
- **2台まで**
- **7日ごとオンライン確認**
- **14日猶予（Grace）**

### 5.2 失効時の挙動（炎上防止）

- Pack生成（有償価値の中心）: **停止**
- 既存成果物の閲覧/索引参照: **可能**
- 復帰導線: 再確認/再ログイン/端末解除をUIで提供

（詳細は `docs/security_and_license.md`）

---

## 6. セキュリティ（不変条件）

### 6.1 外部送信ポリシー

- リポジトリ内容（コード/設定/Pack/ログ）を外部送信しない
- 送信は **ライセンス確認のみ**

### 6.2 読み取り制限

- デフォルトはVS Codeワークスペース配下のみ
- allowlist/denylistで制御可能

### 6.3 secret混入防止（必須）

- 生成前後で二重スキャンし、マスク/除外する
- `.env`、鍵、Terraform state、credential類はデフォルトで除外/強マスク候補

## 7. アーキテクチャ（Core / Front / Plugins）

### 7.1 目標

入口（VS Code/CLI/MCP）が増えても、出力仕様がブレない。

### 7.2 コンポーネント

- **Front**
  - VS Code拡張（主）
  - CLI（保険）
  - MCP（必要に応じて）
- **Core**
  - Pack生成、契約抽出、索引、ルーティング、キャッシュ、決定的出力、セキュリティ、ライセンス
- **Plugins**
  - GCP/IaC、C#（軽量/重解析）、TS、Python、Rust…

（詳細は `docs/architecture_overview.md`）

---

## 8. 解析の二段階（必須）

### 8.1 軽量解析（どこでも動く）

- ファイル構造、依存宣言、参照文字列、設定キー、Terraform/YAML構造など
- C#は構文・using・public API・簡易依存

### 8.2 重解析（揃えば効く）

- 型解決、参照解決、より正確な影響範囲

### 8.3 不変条件

- どんな環境でも **軽量解析へフォールバックして動き続ける**

---

## 9. ルーティング（言語指定を不要にする）

原則として `--lang` をユーザーに強制しない。Coreがワークスペースを観察し自動選択する。

例（検出ヒント）:
- Terraform: `*.tf`
- Cloud Build: `cloudbuild.yaml`
- C#: `*.sln` / `*.csproj`
- TypeScript: `package.json` / `tsconfig.json`
- Python: `pyproject.toml`
- Rust: `Cargo.toml`

曖昧な場合は候補提示し、Frontで選択可能にする。

---

## 10. 出力仕様（IR & Context Pack）

### 10.1 entry（入口）

Phase 1は file/symbol 両対応。ユーザー入力は文字列1本で良い。

- `file:<path>`
- `symbol:<qualifiedName>`
- `auto:<text>`（曖昧なら候補返し）

### 10.2 budget（太さ制御）

Phase 1から必須。Packが太り過ぎてAI運用が破綻しないための装置。

推奨:
- `maxFiles`
- `maxTotalLines` or `maxTotalBytes`
- `maxSnippetsPerFile`

### 10.3 Pack（Markdown）の最小構造

1. goal
2. entry と解決結果（ResolvedEntry）
3. 該当契約（Contracts）
4. 重要索引（影響範囲/依存/参照）
5. 参照ファイル一覧（理由、優先度）
6. secret対策サマリ（伏字/除外）
7. Unknown/Candidates

（詳細は `docs/ir_and_pack_spec.md`）

## 11. Phase計画（完全版ロードマップ）

### 11.1 Phase 1（販売開始の最小構成）

目的: VS Codeから“今開いているファイル”起点でPack生成が安定して使える。

必須:
- VS Code拡張: 現在ファイル→Pack生成、ライセンス状態表示
- Core: pack/index/contracts/security/license（最小）
- GCP/IaCプラグイン: Terraform/YAML/Cloud Build/Workflows/IAMの最低限抽出
- C#プラグイン: 軽量解析（全OS）、Windowsで強化余地

（詳細は `docs/scope_phase1.md`）

### 11.2 Phase 2（精度と収束性の向上）

- 影響範囲推定の精度を上げる（参照解析の強化、LSP連携等）
- 失敗ログ圧縮→再Pack生成（ビルド/テスト/リンタの結果を要約）
- ルール検査の拡充（依存境界、危険パターン、ポリシー違反）
- Packの“太さ制御”をより賢く（優先度、確信度、候補提示）

### 11.3 Phase 3（多言語拡張とプラグイン化の完成）

- TS/Python/Rustの解析プラグインを追加（軽量→重解析の段階拡張）
- GCP領域を拡張（IAM/Network/イベント駆動の契約抽出の充実）
- IRのJSON Schema化と後方互換ポリシー確立

### 11.4 Phase 4（配布・運用・収益の拡張）

- 自動アップデート（署名付き配布）
- macOS署名/公証の整備（販売品質）
- 法人向けプランへの拡張（必要になった段階で）
