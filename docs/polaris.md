# Polaris

## 目的

`Rulixa` の中心価値は、AI のための一時的な入力最適化ではなく、設計知の成果物を継続生成し、レビュー・監査・保守に使える形で残し続けることです。  
AI 入力の正規化は重要な機能ですが、それはこの基盤が持つ価値の一部に過ぎません。

Rulixa が継続生成する主な成果物は次の 4 つです。

- `Contracts`
  実装、設定、依存関係、変更理由を説明可能にする設計成果物
- `Index / Map`
  Symbol、設定、影響経路、変更波及を素早く把握するための索引
- `Context Pack`
  AI が変更開始時に必要とする最小文脈束
- `Security Gate`
  secret、危険な設定、レビュー対象差分を事前に検査する境界

## Rulixa が解く課題

- 設計知がコード断片や個人記憶に埋もれ、レビューや保守で再発明が起きる
- PR レビュー時に「どこが影響範囲か」「何を見ればよいか」が毎回探索になる
- 監査や運用で、どの入力からどの成果物が作られたかを追いにくい
- 実装と設計成果物がずれ、ドリフト検知が後手になる
- AI 利用時に必要な文脈が都度ばらばらに集められ、品質が安定しない

## 成果物の見え方

Rulixa が最終的に提供したい運用価値は次です。

1. 設計成果物一式が継続生成されること
2. PR レビューや差分確認に使える成果物が揃うこと
3. 監査証跡とドリフト検知に使える状態を保てること
4. その一部として Context Pack を必要時に取り出せること

つまり Context Pack は重要ですが、Rulixa の主役ではありません。  
Rulixa の本質は、設計・レビュー・監査・運用に使える成果物群を継続的に整備する基盤であることです。

## 原則

- 説明可能性
  後から読んでも、なぜその成果物が生成されたか説明できる
- 最小責務
  変更開始時に必要な文脈だけを切り出し、過不足を減らす
- フィードバック
  生成結果をレビューと運用に戻し、次の改善へつなげる
- 境界管理
  secret、危険設定、レビュー対象外の混入を防ぐ
- ローカル優先
  コードや Pack を外部へ不用意に送らず、必要最小限で運用する

## 実装方針

実装は段階的に進めるが、責務は常に Core に寄せる。

- Frontend
  VS Code / Cursor / CLI / MCP などの入口
- Core
  解析、因果抽出、成果物生成、レビュー支援、ライセンス、キャッシュ
- Plugins
  GCP / IaC / C# / TypeScript / Python / Rust などの対象別抽出

## Phase 1 との関係

Phase 1 の `WPF + .NET 8` 対応は、Rulixa 全体を WPF 専用ツールにするためではなく、実戦投入可能な最初の具体攻略対象として絞っているものです。  
正本は常に `polaris.md` と `project_full_spec.md` であり、`docs/spec/phase1/*` はその下位仕様として扱います。
