## 北極星（Polaris）

このプロジェクトの北極星は、**AI駆動開発の「トークン不足／暗黙仕様／巨大コードベース」を、誰の環境でも再現可能に“攻略可能な作業単位”へ変換する**ことです。

---

## 1. 解決したい本質課題

- **コンテキスト不足**: 仕様・制約・依存関係が広く散らばり、LLMが一度に保持できない
- **暗黙契約の多さ**: UI/設定/IaC/ライブラリ制約などがテキストや慣習に埋もれる
- **再現性の欠如**: 人や環境によって「何を読ませるべきか」「どこを直すべきか」がブレる
- **事故の温床**: 秘密情報、危険なコマンド、誤った依存境界がAIと相性悪い

---

## 2. 北極星（提供価値）

本ツールは、リポジトリから次を **決定的（同入力→同出力）** に生成し、AIに渡す“最小コンテキスト”を標準化します。

- **契約（Contracts）**: UI/API/データ/IaC/外部依存の前提・入出力・禁止事項を抽出し、短い形で固定
- **索引（Index/Map）**: シンボル地図・依存関係・影響範囲を機械的に可視化
- **Context Pack**: 目的（goal）と入口（entry: file/symbol）から、必要最小限のファイル束＋契約＋制約を組み立て
- **安全ガード（Security Gate）**: 秘密情報の検出・マスク、読み取り範囲制限、危険操作の抑止

結果として、AIは「全文を読む」ではなく、**契約＋差分＋検証ログ**で局所問題を解く運用に寄せられます。

---

## 3. 対象（主戦場と優先順位）

- **主戦場**: GCP（Cloud Run / PubSub / Workflows / IAM / Terraform / Cloud Build 等）
- **次点**: C# / TypeScript / Python / Rust のコードベース
- **入口（フロント）**: VS Code（Cursor含む）を主導線に、必要に応じてCLI/MCPも提供

### OS方針（Phase 1）

- **Windows**: フル機能（C#強め＋GCP/IaC）
- **macOS/Linux**: ライト機能（GCP/IaC中心＋C#は軽量解析のみ）
  - C#軽量解析: `using/公開API/簡易依存` まで（型解決・完全ビルド再現は保証しない）

---

## 4. 成功指標（North Star Metrics）

- **Pack有効率**: 生成されたContext Packだけで、AIが「作業開始→変更→ビルド/テスト」へ進める割合
- **修正収束性**: 1つの目的に対し、少ない反復回数で安定して収束する割合（エラー要約→再生成が効いているか）
- **事故率の低さ**: secret混入、危険コマンド誘導、依存境界破壊などの重大事故がゼロに近いこと
- **再現性**: 同じentry/goal/budgetで誰が実行しても同じ成果物が得られること

---

## 5. 設計原則（不変条件）

- **境界中心**: 外部仕様・暗黙情報は「契約」として境界に集約する
- **決定的出力**: 同入力→同出力（差分レビュー可能）を優先する
- **最小化**: Packは常にbudgetに収め、漏れは「候補」として明示する
- **フォールバック**: 高精度解析が無理な環境でも、軽量解析で“動き続ける”
- **安全第一**: 生成物にsecretを含めない／読み取り範囲を限定／外部送信しない（ライセンス確認を除く）

---

## 6. 非目標（やらないこと）

- LLM自体の性能を直接上げる（モデル改善は対象外）
- コード全文の要約を常に提供する（必要な最小コンテキストに集中）
- 100%の型解決やビルド再現を全OSで保証する（必要なら将来拡張）
- リポジトリ内容（コード/pack）をサーバーへ送信する（ライセンス通信以外はしない）

---

## 7. 使い方の基本（この北極星の運用）

- 入口（file/symbol）＋目的（goal）＋太さ（budget）で **Context Packを生成**
- AIはPackの契約に従って差分を作り、**コンパイラ/テスト/リンタ**を審判に反復する
- 失敗ログはツールが **短く分類・圧縮**し、再びPackに反映する
