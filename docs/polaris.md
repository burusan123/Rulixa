# Polaris

## 目的

`Rulixa` の北極星は、AI の性能そのものを上げることではなく、巨大コードベースや複雑な設計知を AI が安全に扱える成果物へ変換し続けることです。

Rulixa は単なる Context Pack 生成ツールではありません。中心価値は次の 4 点です。

- `Contracts`
  仕様、依存、禁止事項、入出力、境界を明文化する
- `Index / Map`
  シンボル、依存、影響範囲、変更導線を見える化する
- `Context Pack`
  AI がその時点の変更に必要な最小文脈を決定的に受け取れるようにする
- `Security Gate`
  secret、危険操作、読み取り範囲、外部送信を制御する

## Rulixa が解く問題

- LLM は巨大コードベース全体を一度に安全に保持できない
- 暗黙仕様が文章、慣習、レビューコメント、設定ファイルに分散している
- 人や入口によって AI に渡す文脈がぶれる
- secret 混入や危険操作など、AI 導入特有の事故が起きる

## 成果物の優先順位

Rulixa が継続生成すべき成果物の優先順位は次です。

1. 設計知の成果物一式
2. PR レビューや監査に使える差分情報
3. ドリフト検知に使える比較可能な出力
4. その一部としての Context Pack

つまり、Context Pack は重要な出力の一つですが、製品全体の主役ではありません。
Rulixa の本質は、設計知・依存・制約・安全要件を継続的に生成し、レビュー可能な形で保持する運用基盤にあります。

## 不変条件

- 決定性
  同じ入力なら同じ成果物が出る
- 最小化
  必要最小限の文脈を返す
- フォールバック
  完全解析できなくても、使える最低限の結果を返す
- 安全第一
  secret 混入、危険操作、許可外読み取りを防ぐ
- ローカル優先
  コードや Pack を外部に送らなくても成立する

## 実装戦略

入口は複数あってよいですが、価値の中心は常に Core に置きます。

- Frontend
  VS Code / Cursor / CLI / MCP などの入口
- Core
  解析、契約生成、Index 生成、Pack 生成、ライセンス、キャッシュ
- Plugins
  GCP / IaC / C# / TypeScript / Python / Rust などの具体解析

## Phase 1 との関係

Phase 1 の `WPF + .NET 8` 対応は、Rulixa 全体を WPF 特化にするためではなく、最初に実戦投入できる攻略対象を選んだ結果です。
製品全体の正本はこの `polaris.md` と `project_full_spec.md` に置き、Phase 1 文書はその具体攻略対象を定義する補助文書として扱います。
